# CONTEXTO — JusArtificial Plataforma SaaS jurídica com IA para advogados. Stack: Backend : Node.js + Express.js (Monolito Modular) Frontend : React 19 + Vite + TailwindCSS IA : Google Gemini API (@google/genai) — Flash e Pro RAG : contextRanker.js (ATUAL: keyword simples — NÃO é RAG semântico) Vectors : vectorStore.ts (ATUAL: localStorage do browser — INVIÁVEL) Validator: validator.ts (ATUAL: regex/listas estáticas — NÃO valida leis reais) OCR : pdf-parse / Tesseract O sistema tem prompts excelentes (Chain of Thought, Personas, Few-Shot), mas a camada de recuperação de contexto (RAG) e validação jurídica são completamente simuladas. Isso expõe advogados ao risco de peças com leis revogadas ou inexistentes — risco profissional e sanção da OAB. ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ # MISSÃO (Codex) Você é o engenheiro de IA e produto responsável por transformar sistemas simulados em implementações reais e seguras. Foco: RAG semântico com embeddings reais, vector store server-side, validação jurídica com APIs oficiais, controle de custo de IA e módulos essenciais de produto. ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ # ITENS ATRIBUÍDOS AO CODEX ━━━━━━ 🔴 CRÍTICO P0 — INTEGRIDADE JURÍDICA ━━━━━━ ### C01 — Implementar RAG Semântico Real CONTEXTO: contextRanker.js usa query.includes(term) — busca por string simples. "Veículo" não encontra documentos com "automóvel". "Réu" não encontra "acusado". Sem contexto semântico relevante, o Gemini alucina e cita leis erradas. ARQUIVO ALVO: backend/services/contextRanker.js (substituir completamente) TAREFA: 1. Criar backend/services/embeddings.js: - Função embedText(text): Promise - Usar @google/genai modelo text-embedding-004 (Google Gecko). - Alternativa aceita: openai SDK com text-embedding-3-small. 2. Criar backend/services/vectorStore.js (server-side): - storeChunks(chunks, userId, firmId, documentId): salva vetores no BD. - searchSimilar(queryVector, userId, firmId, topK=5): busca por cosseno. - Fase 1 MVP: similaridade de cosseno em memória (aceitável até ~500 chunks). - Fase 2 produção: integrar Pinecone SDK ou pgvector. 3. Pipeline de ingestão de documentos (chamado após OCR): texto bruto → chunking (512 tokens, 50 tokens overlap) → embedding por chunk → armazenar (chunkId, firmId, userId, text, vector) 4. Pipeline de retrieval (chamado no chat): mensagem do usuário → embed query → busca por similaridade → top-5 chunks → injetar como contexto no prompt do Gemini 5. contextRanker.js antigo: SUBSTITUÍDO, não expandido. ENTREGÁVEL: - backend/services/embeddings.js com embedText() + JSDoc. - backend/services/vectorStore.js com storeChunks() e searchSimilar(). - backend/services/contextRanker.js refatorado usando RAG semântico. - Teste: "veículo" encontra documentos com "automóvel" (similaridade > 0.7). - Documentação do modelo de embedding e custo estimado por query. CRITÉRIO: Recall semântico > 80% em queries jurídicas com sinônimos. Zero keyword matching no fluxo de RAG. ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ ### C02 — Migrar vectorStore de localStorage para servidor CONTEXTO: services/vectorStore.ts (FRONTEND) salva embeddings no localStorage. Limite do browser: ~5MB. Um escritório com 10 processos de 50 páginas cada = ~50MB de texto embedado → localStorage estoura → app quebra silenciosamente. ARQUIVO ALVO: services/vectorStore.ts (frontend) — DELETAR TAREFA: 1. Deletar COMPLETAMENTE services/vectorStore.ts do frontend. 2. Frontend nunca armazena embeddings ou vetores. 3. Acesso a vetores via POST /api/chat → backend faz o retrieval (C01). 4. Limpar dados residuais no localStorage dos usuários existentes: Object.keys(localStorage) .filter(k => k.startsWith('vector_') || k.startsWith('embed_')) .forEach(k => localStorage.removeItem(k)); 5. Documentar migração em MIGRATION.md. ENTREGÁVEL: - services/vectorStore.ts deletado. - Script de limpeza de localStorage para usuários existentes. - MIGRATION.md explicando a mudança e impacto. CRITÉRIO: Zero embedding em localStorage. Sem limite de tamanho para documentos. ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ ### C03 — Validador de Citações Jurídicas Real CONTEXTO: services/validator.ts usa regex simples como "Lei > 20000 é suspeita". O sistema pode aprovar leis revogadas ou completamente inexistentes. Um advogado com peça contendo lei errada pode perder o processo + sanção da OAB. ARQUIVO ALVO: services/validator.ts → backend/services/legalValidator.js TAREFA: 1. Criar backend/services/legalValidator.js integrando APIs reais: OPÇÃO A — LexML.gov.br (API pública e gratuita): GET https://www.lexml.gov.br/urn/urn:lex:br:federal:lei:{ano};{numero} Retorna: vigência, data de revogação, ementa da lei. OPÇÃO B — Planalto.gov.br (scraping estruturado da legislação federal): Verificar existência e vigência da norma na base oficial. OPÇÃO C — Jusbrasil/Digesto API (para planos enterprise): Documentar configuração via JUSBRASIL_API_KEY no .env. 2. Fluxo de validação: Gemini gera peça → extrair todas as referências a leis/artigos → para cada citação: consultar API → verificar existência e vigência → retornar: { citation, valid, status: 'vigente'|'revogada'|'not_found', explanation, suggestedReplacement } 3. Cache de 24h dos resultados (leis raramente mudam diariamente). 4. Regra crítica: lei não verificável (timeout, API fora) → status: 'não_verificada' NUNCA marcar como 'válida' por omissão. 5. Frontend: citações inválidas marcadas em vermelho inline na peça gerada, com tooltip da explanation e botão "Substituir" sugerindo a lei correta. ENTREGÁVEL: - backend/services/legalValidator.js com ao menos 1 API real integrada. - Cache de resultados no Firestore ou Redis. - Endpoint POST /api/validate/citations recebendo array de citações. - Testes: Lei 8.666/93 → status: 'revogada' (revogada pela Lei 14.133/2021). Lei 14.133/2021 → status: 'vigente'. Lei 12.999/2099 → status: 'not_found'. - Frontend: marcação visual de citações inválidas + tooltip + botão substituir. CRITÉRIO: Zero peça aprovada com lei revogada ou inexistente. Validação real, não heurística. Latência < 3s para validar até 10 citações. ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ ### C04 — Google Search real no Gemini (eliminar alucinação de busca) CONTEXTO: Os prompts instruem o modelo a usar "Busca do Google", mas se a ferramenta googleSearch não for injetada na chamada à API, o modelo ALUCINA que pesquisou — inventa resultados fictícios de jurisprudência. ARQUIVO ALVO: backend/services/aiOrchestrator.js (criado pelo Jules J03) TAREFA: 1. Injetar ferramenta google_search em todas as chamadas de chat Pro/Enterprise: const response = await genAI.models.generateContent({ model: 'gemini-1.5-pro', contents: [...messages], tools: [{ googleSearch: {} }], config: { temperature: 0.2 } }); 2. Processar groundingMetadata da resposta: - Extrair fontes consultadas (STJ, STF, TRFs, Planalto). - Incluir no retorno ao frontend: sources: [{ title, url, snippet }]. 3. Frontend exibe as fontes abaixo de cada resposta para o advogado verificar. 4. Toggle por plano: - Free: sem busca web (usa apenas contexto dos documentos uploaded). - Pro e Enterprise: busca web habilitada. ENTREGÁVEL: - Gemini com googleSearch tool injetada para planos Pro/Enterprise. - Frontend exibe sources após resposta. - Testes: resposta com groundingMetadata inclui sources no response body. CRITÉRIO: Zero "alucinação de busca". Modelo ou usa a ferramenta real ou declara explicitamente que não pode buscar — nunca inventa resultados. ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ ━━━━━━ 🟡 P1 — PRODUTO VENDÁVEL ━━━━━━ ### C05 — Model Router: Flash vs Pro (controle de custo) CONTEXTO: Sem roteamento de modelo, o Critic Loop pode triplicar chamadas ao Pro. Um contrato complexo pode custar > $0.50 USD por execução. Para SaaS em BRL com margem estreita, isso é insustentável. TAREFA: 1. Criar backend/services/modelRouter.js: Usar Gemini Flash (barato): - Resumos de documentos curtos (< 5 páginas) - Perguntas simples de contexto - Extração de metadados de processos - RAG retrieval (embedding + ranqueamento) Usar Gemini Pro (caro, alta qualidade): - Geração de peças processuais completas - Análise jurídica complexa - Critic Loop (apenas para geração de peças) - Contratos com mais de 10 páginas 2. Critic Loop: disparado APENAS para geração de peças — não para chat simples. 3. Budget diário por plano: - Free: 10 chamadas Pro/dia, 50 Flash/dia. - Pro: 50 Pro/dia, Flash ilimitado. - Enterprise: sem limite. 4. Endpoint GET /api/usage/today com tokens usados e custo estimado. ENTREGÁVEL: - backend/services/modelRouter.js com roteamento documentado. - Budget por plano implementado no Firestore. - Endpoint /api/usage/today. CRITÉRIO: Custo de chat simples reduzido ≥ 70% usando Flash. Pro reservado para geração de peças — Critic Loop não duplica custo no chat. ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ ### C06 — Padronizar erros da API com RFC 7807 CONTEXTO: Erros retornam formatos inconsistentes — string, objeto, código HTTP errado.
