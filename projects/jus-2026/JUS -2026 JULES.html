Backend/Segurança: 20% pronto. Infra/Escala: 10% pronta. Seu foco exclusivo: segurança crítica, autenticação, IDOR, multitenancy, infraestrutura robusta e conformidade com LGPD. ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ # ITENS ATRIBUÍDOS AO JULES ━━━━━━ 🔴 CRÍTICO P0 — SEGURANÇA IMEDIATA ━━━━━━ ### J01 — Ativar autenticação em 100% das rotas /api CONTEXTO: server.js roda em modo "demo" com auth não forçada em rotas como /api/sessions e /api/documents. Qualquer pessoa sem token acessa a API. ARQUIVO ALVO: backend/server.js · backend/routes/*.js TAREFA: 1. Remover TODA lógica condicional de "demo mode" que bypassa auth. 2. Aplicar middleware `authenticate` (authMiddleware.js) em TODAS as rotas /api/*. 3. Única rota pública permitida: /health (uptime check). 4. Teste de integração verificando que requisições sem token retornam 401. ENTREGÁVEL: - backend/server.js sem bypass de demo. - router.use(authenticate) no início de cada router. - tests/auth.integration.test.js com 5 casos: sem token, token expirado, token inválido, token válido, rota pública. CRITÉRIO: Requisição sem token em qualquer /api/* → 401. Zero exceção. Zero bypass. ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ ### J02 — Corrigir IDOR em sessões e documentos CONTEXTO: backend/sessions.js e sessionRoutes.js aceitam userId/IDs enviados pelo CLIENTE sem verificar se o recurso pertence ao usuário autenticado. Qualquer usuário autenticado pode listar/deletar sessões de outro usuário. ARQUIVO ALVO: backend/sessions.js · backend/routes/sessionRoutes.js backend/routes/documentRoutes.js TAREFA: 1. Em TODAS as operações de leitura/escrita/deleção: - IGNORAR userId vindo de req.params, req.body ou req.query. - USAR SEMPRE req.user.uid extraído pelo middleware de autenticação. 2. Verificação obrigatória de propriedade antes de qualquer operação: const resource = await getResource(resourceId); if (!resource || resource.ownerId !== req.user.uid) { return res.status(403).json({ error: 'Acesso negado' }); } 3. Aplicar mesmo padrão para firmId quando multitenancy estiver ativo (J04). ENTREGÁVEL: - Todos os controllers com verificação de propriedade. - tests/idor.security.test.js com cenários: usuário A acessa recurso do B (403), acessa próprio (200), ID inexistente (404). CRITÉRIO: Nenhum recurso acessível por UID diferente do dono. Zero IDOR. ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ ### J03 — Migrar orquestração de IA para o backend CONTEXTO: services/gemini.ts no FRONTEND contém Chain of Thought, Critic Loop, fallback de modelo, retry e parâmetros de temperatura. Um usuário avançado pode manipular o comportamento do agente via DevTools. ARQUIVO ALVO: services/gemini.ts (frontend) → backend/services/aiOrchestrator.js backend/routes/chatRoutes.js TAREFA: 1. Criar backend/services/aiOrchestrator.js com toda a lógica de gemini.ts: - Retry com backoff exponencial (máx 3 tentativas). - Fallback: Gemini Pro falhou → Gemini Flash. - Critic Loop: validação server-side após geração. - Temperatura, system prompts e configuração do modelo: NUNCA no frontend. 2. Frontend (services/gemini.ts) → thin client apenas com POST /api/chat. 3. System prompts ficam em backend/data/prompts.js (nunca no bundle do cliente). ENTREGÁVEL: - backend/services/aiOrchestrator.js com orquestração completa + JSDoc. - services/gemini.ts reduzido a thin client (< 50 linhas). - Testes: fallback funciona, retry respeita backoff exponencial. CRITÉRIO: Zero lógica de negócio de IA no frontend. System prompts não aparecem no bundle. ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ ━━━━━━ 🟡 P1 — PRÉ-VENDA ENTERPRISE ━━━━━━ ### J04 — Implementar Multitenancy real com firmId (Escritório) CONTEXTO: Banco usa apenas userId como partição. Não existe conceito de firmId. Advogados do mesmo escritório não podem colaborar com isolamento garantido. Se um advogado sair, seus dados ficam órfãos. TAREFA: 1. Criar collection firms no Firestore: firms/{firmId}: name, planId, createdAt, ownerId. 2. Adicionar firmId em TODAS as collections: sessions, documents, auditLogs. 3. Criar middleware resolveTenant: - Lê firmId do token customizado ou perfil do usuário. - Injeta req.firmId em todas as rotas. 4. Todas as queries com filtro duplo: firmId + userId. 5. Endpoints: POST /api/firms · POST /api/firms/invite. ENTREGÁVEL: - Script de migration para adicionar firmId a dados existentes. - middleware/resolveTenant.js. - Todos os controllers com filtro firmId. - Testes: advogado da firma A não acessa dados da firma B. CRITÉRIO: Zero cross-tenant data leak. Dados isolados por firmId em 100% das queries. ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ ### J05 — Migrar upload para Stream (eliminar memoryStorage) CONTEXTO: server.js usa multer.memoryStorage(). 100 uploads simultâneos de 10MB = 1GB de RAM instantaneamente + OCR = crash por OOM (Out of Memory). ARQUIVO ALVO: backend/server.js · backend/routes/uploadRoutes.js TAREFA: 1. Substituir memoryStorage por stream direto para Firebase Storage via busboy. Arquivo nunca fica inteiramente em memória do servidor. 2. OCR como job assíncrono: - Upload retorna imediatamente com { jobId }. - OCR processado em background (Firestore Queue ou Cloud Tasks). - Status via GET /api/uploads/:jobId/status. 3. Limite de tamanho de arquivo: 30MB por upload. ENTREGÁVEL: - Upload refatorado sem memoryStorage. - OCR como worker assíncrono. - tests/upload.load.test.js simulando 20 uploads simultâneos sem OOM. CRITÉRIO: Upload de 20MB não aloca > 2MB de RAM no processo Node.js. ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ ### J06 — Redis Adapter para Socket.io (escala horizontal) CONTEXTO: WebSocket roda na mesma instância da API. Sem Redis Adapter, escalar horizontalmente fará usuários em instâncias diferentes não se comunicarem. TAREFA: 1. Instalar @socket.io/redis-adapter + ioredis. 2. Configurar adapter com pubClient + subClient. 3. Extrair config de Socket.io para backend/services/realtimeService.js. 4. REDIS_URL em .env.example + validação de env no boot. 5. Sem Redis configurado: fallback para modo single-instance (dev). ENTREGÁVEL: - Socket.io com Redis Adapter. - backend/services/realtimeService.js isolado. - docker-compose.yml com Redis para dev local. CRITÉRIO: 2 instâncias do backend + Redis → usuários em instâncias diferentes se comunicam. Zero mensagens perdidas entre instâncias. ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ ### J07 — Migrar compliance LGPD para o backend CONTEXTO: services/compliance.ts (FRONTEND) chama SDK do Firebase diretamente para deletar dados do usuário. Se regras do Firestore não estiverem corretas, um usuário malicioso pode limpar dados de outros. TAREFA: 1. Criar backend/routes/complianceRoutes.js: - DELETE /api/compliance/wipe — apaga TODOS os dados do usuário autenticado. - GET /api/compliance/export — exporta todos os dados em JSON (portabilidade LGPD). 2. Ambas auditadas no audit.js antes de executar. 3. Deleção cobre: sessões, documentos, logs de chat, preferências, embeddings. 4. Remover wipeClientData do frontend (nunca chamar Firestore diretamente para ops destrutivas). ENTREGÁVEL: - backend/routes/complianceRoutes.js com as 2 rotas. - Frontend atualizado para chamar backend (não SDK diretamente). - Runbook: SLA de deleção máx 30 dias conforme LGPD. CRITÉRIO: Wipe e export 100% server-side. Frontend nunca chama Firestore diretamente para operações de dados sensíveis. ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ ### J08 — Sanitizar audit logs para conformidade LGPD CONTEXTO: backend/audit.js grava logs no Firestore. Se o campo details gravar prompts do usuário, dados pessoais de clientes dos advogados (terceiros) serão persistidos sem consentimento — violação direta da LGPD. TAREFA: 1. Auditar todos os campos gravados por audit.js. 2. Nunca gravar: conteúdo de mensagens, texto de documentos, nomes de partes. 3. Logs apenas: userId, firmId, action, resourceId, timestamp, ipHash (hash, não IP bruto). 4. Política de retenção: logs > 2 anos deletados automaticamente (Cloud Function). 5. docs/lgpd-audit-policy.md documentando o que é e não é gravado. ENTREGÁVEL: - audit.js refatorado com schema de campos permitidos. - Cloud Function de limpeza de logs antigos. - docs/lgpd-audit-policy.md. - Testes: mensagem do usuário não aparece no log. CRITÉRIO: Zero dado pessoal de clientes em logs. Retenção automatizada. ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ ━━━━━━ 🟢 P2 — ESCALA E QUALIDADE ━━━━━━ ### J09 — Rate Limiting real por usuário e escritório CONTEXTO: Rate limit atual é "permissivo para demo". Um único usuário ou ataque pode exaurir a quota da Gemini API (custo) ou derrubar o servidor. TAREFA: 1. Instalar express-rate-limit + rate-limit-redis. 2. Limites por camada: - Global: 300 req/min por IP. - Autenticado: 100 req/min por userId. - Chat (IA): 20 mensagens/min por userId — protege custo da Gemini. - Upload: 5 arquivos/min por userId. 3. Resposta 429 com Retry-After e body RFC 7807. 4. Limites configuráveis via env vars. CRITÉRIO: Quota da Gemini protegida. DDoS por usuário bloqueado em < 1 segundo. ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ ### J10 — CI/CD com quality gates de segurança CONTEXTO: Código vai direto para produção sem verificação de vulnerabilidades, cobertura de testes ou análise de segurança estática.
